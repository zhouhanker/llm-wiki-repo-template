# LLM Wiki Repo Template

一个面向人类与 AI Agent 协作的轻量级知识库模板。它用于把原始资料、稳定知识、项目状态和临时交付分层保存，让不同会话中的 Agent 能快速理解上下文、延续任务，并留下可验证的工作记录。

## 适用场景

- 为一个新项目快速建立可持续维护的知识库
- 汇总文档、网页、访谈记录等原始资料
- 沉淀概念、实践模式与项目决策
- 在多轮或多 Agent 协作中保留任务上下文
- 用计划、风险与验证证据管理长期任务

## 当前文件

| 文件 | 用途 |
| --- | --- |
| [`schema.md`](./schema.md) | 定义 Source、Concept、Pattern 和 Project 四类 Wiki 页面的最低字段与写入规则 |
| [`agent.md`](./agent.md) | 按时间记录用户需求、任务拆解和 Agent 操作上下文，便于后续会话接续 |
| [`fplan.md`](./fplan.md) | 维护项目目标、调研结论、参考资料、分阶段计划、执行规则和目标架构 |
| [`warn.md`](./warn.md) | 按严重程度登记风险，并维护项目验证关卡 |

除 `schema.md` 外，其余文件包含待替换的占位符。使用模板时，应先将 `$replace current project name$`、`${timestamp}` 和 `${Fill in the actual content}` 替换为项目的真实信息。

## 推荐目录结构

仓库初始只提供核心模板文件。可根据项目需要创建以下目录：

```text
.
├── README.md       # 项目入口和使用说明
├── schema.md       # Wiki 页面规范
├── agent.md        # Agent 任务与上下文记录
├── fplan.md        # 分阶段实施计划
├── warn.md         # 风险登记与验证关卡
├── raw/            # 原始资料与证据
├── wiki/           # 已提炼、可复用的稳定知识
└── output/         # 报告、草稿等临时交付物
```

目录职责遵循以下边界：

- `raw/` 保存原始内容，尽量保留来源地址、作者和获取时间。
- `wiki/` 保存经过整理且能够长期复用的结论。
- `output/` 保存阶段性产物，不默认视为稳定知识。
- 项目级状态、决策和下一步写入 Project 页面或 `fplan.md`。

## 快速开始

1. 复制本仓库作为新项目的知识库。
2. 在 `agent.md`、`fplan.md` 和 `warn.md` 中替换项目名称及全部占位符。
3. 按实际需要创建 `raw/`、`wiki/` 和 `output/` 目录。
4. 在 `fplan.md` 中填写总体目标、阶段任务和验收条件。
5. 在 `warn.md` 中登记已知风险，并为关键结果设置验证关卡。
6. 依据 `schema.md` 创建第一批 Source、Concept、Pattern 或 Project 页面。

建议在初始化后检查是否仍有未处理的占位符：

```bash
rg -n '\$replace|\$\{Fill in|\$\{timestamp\}' .
```

## Wiki 页面类型

### Source

记录资料的来源、摘要、关键观点和可复用结论。Source 页面是其他知识页的证据入口，应能追溯到原文或外部来源。

### Concept

解释一个概念的定义、边界、适用范围和常见误解。重要结论应引用对应的 Source 页面。

### Pattern

沉淀可重复使用的流程、模板、检查清单或实践方法，重点说明适用场景、操作步骤、验收标准和风险。

### Project

保存具体项目的当前目标、进展、阻塞项、关键决策、验证证据和下一步，确保新会话可以快速接手。

各类型的完整字段模板见 [`schema.md`](./schema.md)。

## 推荐工作流

```text
收集资料 → 写入 raw/ → 创建 Source 页面 → 提炼 Concept / Pattern
                                      ↓
                             更新 Project 页面
                                      ↓
                         同步计划、风险与验证证据
```

一次典型的 Agent 协作流程如下：

1. 开始任务前，阅读 `README.md`、`schema.md`、`fplan.md` 和 `warn.md`。
2. 在 `agent.md` 记录用户原始需求、约束条件和当前操作上下文。
3. 按 `fplan.md` 执行当前阶段，并持续保存可复核的证据。
4. 将原始输入保留在 `raw/`，把稳定结论整理到 `wiki/`。
5. 将报告、导出文件或一次性交付放入 `output/`。
6. 完成阶段任务后更新计划状态、风险项、验证结果和下一步。

## 内容维护约定

- 每个 Wiki 页面至少包含页面类型、当前状态、来源、更新时间、摘要、相关页面和待验证问题。
- 事实、推断和待验证内容应明确区分；无法确认的结论不要写成既定事实。
- 关键决策应同时记录原因、影响范围与验证证据。
- 页面之间使用相对链接，避免知识成为无法导航的孤岛。
- 内容发生实质变化时更新日期；过时内容应标注状态，而不是直接丢失历史依据。
- 阶段完成以验证证据为准，不只以任务勾选状态为准。

## 完成标准

一个初始化完成的项目至少应满足：

- 核心模板文件中没有无意保留的占位符。
- `fplan.md` 有清晰的总体目标、当前阶段和验收条件。
- `warn.md` 已覆盖当前已知的关键风险与验证关卡。
- 原始资料可以追溯来源，稳定结论可以追溯到资料或验证证据。
- Project 页面或等价记录包含当前状态、阻塞项和明确的下一步。
- 新加入的人或 Agent 仅阅读仓库文档即可继续工作。

## 保持模板轻量

本项目不绑定特定的 Wiki 引擎、静态站点生成器或 Agent 框架。Markdown 文件是唯一基础依赖；搜索可使用 `rg`，版本管理可使用 Git。只有在检索、发布或协作规模确实需要时，再引入索引、自动化脚本或站点构建工具。
